#### 🔴 문제 3: 분류 기준 불명확

##### 3-1. 고객 관련 기능 분산

| 기능 | 현재 위치 | 논리적 위치 | 문제점 |
|------|----------|------------|--------|

| 고객 리뷰 (리뷰 조회) | 고객 인사이트 | ❌ 중복 | 리뷰 분석과 기능 중복 |

**문제**: "고객 인사이트"의 "고객 리뷰"와 "분석 > 고객 분석 허브 > 리뷰 분석"이 기능적으로 중복

##### 3-2. 분석 영역의 "기타" 분류

| 기능 | 현재 위치 | 논리적 위치 | 문제점 |
|------|----------|------------|--------|
| 작가 분석 | 분석 > 기타 | ⚠️ 모호 | 작가 성과 분석 → 성과 허브? 작가 관리 → 별도 카테고리? |
| 비용 & 손익 | 분석 > 기타 | ⚠️ 모호 | 재무 분석 → 별도 카테고리? 성과 분석과 통합? |

**문제**: "기타" 분류로 인해 확장성 및 논리적 그룹핑 어려움

##### 3-3. Business Brain의 역할 모호

| 기능 | Business Brain 내부 | 다른 페이지와 중복? | 문제점 |
|------|:------------------:|:-----------------:|--------|
| RFM 분석 | ✅ 있음 | 고객 분석과 중복 | 동일 기능이 두 곳에 존재 |
| 코호트 분석 | ✅ 있음 | 고객 분석과 중복 | 동일 기능이 두 곳에 존재 |
| 매출 트렌드 | ✅ 있음 | 성과 분석과 중복 | 유사 차트 및 지표 |
| 작가 건강도 | ✅ 있음 | 작가 분석과 중복 | 유사 지표 |
| 쿠폰 분석 | ✅ 있음 | 쿠폰 효과 분석과 중복 | 유사 지표 |

**문제**: Business Brain이 "종합 대시보드"인지 "고급 분석 도구"인지 역할 불명확

#### 🔴 문제 4: 1차 데이터 중심 구조

**현재 데이터 흐름**:
```
Raw Data (Google Sheets)
    ↓
각 페이지에서 개별 조회/가공
    ↓
페이지별 독립적 계산
    ↓
화면 표시
```

**문제점**:
- 동일 raw data를 각 페이지에서 개별 가공 → 재사용 불가
- 월별 GMV 계산 로직이 4곳에 분산 → 일관성 보장 어려움
- 확장 시 중복 코드 증가

**예시: 월별 GMV 계산**
- `대시보드`: `daily_metrics` 테이블에서 집계
- `성과 분석`: raw `order` 데이터에서 직접 집계
- `주문 패턴 분석`: `order` 데이터에서 월별 집계
- `Business Brain`: 별도 캐시에서 집계

→ **4가지 다른 방식으로 동일 지표 계산**

## 2. 개편 방향 (To-Be 원칙)

### 2.1 메뉴 개편 원칙

#### 원칙 1: "무엇을 관리하는가 / 무엇을 분석하는가" 기준 재분류

| 기준 | 설명 | 예시 |
|------|------|------|
| **관리 (Operation)** | 일상 업무 실행 및 모니터링 | 물류 운영, 업무 지원 |
| **분석 (Analysis)** | 데이터 기반 인사이트 도출 | 성과 분석, 고객 분석 |
| **인사이트 (Insight)** | 전략적 의사결정 지원 | Business Brain, 예측 분석 |
| **도구 (Tools)** | 작업 실행 및 자동화 | 쿠폰 생성, AI 어시스턴트 |

#### 원칙 2: Raw Data → 가공 Data → 인사이트 흐름 구조화

```
L0: Raw Data (Google Sheets)
    ↓
L1: 1차 가공 데이터 (daily_metrics, aggregated tables)
    ↓
L2: 2차 가공 데이터 (analytics_coupon, analytics_customer)
    ↓
L3: 인사이트 (Business Brain, 예측 모델)
```

**메뉴 구조 반영**:
- **관리/운영**: L0 데이터 직접 조회
- **분석**: L1/L2 데이터 활용
- **인사이트**: L2/L3 데이터 기반 전략 제안

#### 원칙 3: 중복 지표는 단일 Source Page로 통합

| 지표 | Source Page | 다른 페이지에서의 역할 |
|------|------------|---------------------|
| **GMV, 주문, AOV** | 성과 분석 | 대시보드: 요약만 표시 |
| **월별 트렌드** | 성과 분석 | Business Brain: 인사이트만 표시 |
| **RFM, 코호트** | 고객 분석 | Business Brain: 요약만 표시 |

#### 원칙 4: 사이드바는 기능 허브, 세부 지표는 하위 탭/필터로 이동

**Before (현재)**:
```
사이드바
├── 성과 분석
├── 주문 패턴 분석
├── 쿠폰 효과 분석
└── ...
```

**After (개편안)**:
```
사이드바
└── 성과 분석 허브
    └── (페이지 내부)
        ├── 탭: 개요
        ├── 탭: 주문 패턴
        ├── 탭: 쿠폰 효과
        └── ...
```

### 3.2 개편안 상세 설명

#### 3.2.1 성과 분석 허브 통합

**Before (현재)**:
```
분석
├── 성과 분석 허브
│   ├── 성과 분석
│   ├── 주문 패턴 분석
│   └── 쿠폰 효과 분석
```

**After (개편안)**:
```
성과 분석 허브
└── 성과 분석 (단일 페이지, 내부 탭 구조)
    ├── 탭: 개요
    │   ├── KPI 카드 (GMV, 주문, AOV, 고객 수)
    │   ├── 월별 트렌드 차트
    │   ├── 국가별 성과
    │   └── 채널별 성과
    │
    ├── 탭: 주문 패턴
    │   ├── 요일별 패턴
    │   ├── 월별/계절성 트렌드
    │   ├── 국가별 패턴
    │   └── 히트맵 (요일 × 월)
    │
    ├── 탭: 쿠폰 효과
    │   ├── 쿠폰 ROI
    │   ├── 전환율 분석
    │   ├── 타입별 성과
    │   └── 국가별 효과
    │
    └── 탭: 채널 분석
        ├── 플랫폼별 성과
        ├── PG별 성과
        └── 결제수단별 성과
```

**통합 근거**:
1. **데이터 소스 동일**: 모두 `order` + `coupon` 데이터 기반
2. **사용자 워크플로우**: "성과 확인 → 패턴 파악 → 쿠폰 효과 확인" 순서로 자연스러움
3. **중복 제거**: 월별 GMV, 주문 건수 등 공통 지표를 단일 페이지에서 관리

**구현 방식**:
- 기존 `/analytics` 페이지를 메인으로 유지
- `/order-patterns`, `/coupon-analytics`는 `/analytics`의 하위 탭으로 통합
- URL은 `/analytics?tab=order-patterns` 형태로 리다이렉트

## 9. 통합 디자인/UX 가이드 (구현 관점)

> **목적**: 메뉴 개편 이후에도 현재 허브와 동일한 **디자인 컴포넌트/페이지 규격**을 유지하면서,  \
> 사용자 입장에서 **혼동 없이** 자연스럽게 탐색할 수 있도록 구현 가이드를 정의

### 9.1 공통 페이지 레이아웃 규격

#### 9.1.1 레이아웃 기본 규칙

- **최상위 구조**
  - 상단: 글로벌 헤더(없다면 각 페이지 헤더가 시각적 앵커 역할)
  - 좌측: 사이드바 (`Sidebar.tsx`) – **허브/카테고리 단위** 탐색
  - 우측 메인: 페이지 컨텐츠 – **페이지 내부 탭/필터/카드** 구성

- **페이지 공통 마진/패딩**
  - 상단 여백: `mt-4`~`mt-6` 고정
  - 좌우 패딩: 데스크톱 기준 `px-4`~`px-6` 일관 유지
  - 카드 간 간격: `gap-4` 또는 `gap-6` 사용 (페이지 전역에서 한 가지 규격 선택)

- **콘텐츠 폭**
  - 기본: 3~4열 그리드 (`grid-cols-1 md:grid-cols-2 lg:grid-cols-3/4`) 패턴 재사용
  - Business Brain처럼 정보 밀도가 높은 페이지도 **그리드 단위**는 동일하게 유지

#### 9.1.2 페이지 헤더 패턴 통일

모든 주요 분석/허브 페이지는 다음 패턴을 공통적으로 사용:

- **구성 요소**
  - 좌측: 아이콘 + 페이지 타이틀 + 서브 타이틀
  - 우측(선택): 요약 배지/태그(예: 기간, 데이터 신뢰도)
- **스타일**
  - `rounded-2xl`, `p-4 lg:p-6`, `shadow-sm` 또는 `shadow-lg`
  - 카테고리별 **강조 색상(그라데이션/배경)**만 다르게 설정

예시 (허브/메뉴별 권장 헤더 색상):

| 메뉴/허브              | 역할                       | 권장 색상/스타일 예시                                                   |
|------------------------|----------------------------|-------------------------------------------------------------------------|
| **홈 (대시보드)**      | 전체 현황 요약             | `bg-idus-500` + 옅은 오렌지 그라데이션                                  |
| **물류 운영**          | 운영 모니터링              | 그린/티얼 계열 `bg-gradient-to-r from-emerald-500 to-teal-500`         |
| **업무 지원**          | 운영 보조 도구             | 뉴트럴 블루/그레이 `bg-gradient-to-r from-slate-600 to-slate-800`      |
| **성과 분석 허브**     | 매출/주문/채널 성과 분석   | 보라/인디고 `bg-gradient-to-r from-violet-600 to-indigo-600`           |
| **고객 인사이트 허브** | 고객/리뷰/여정 인사이트    | 블루/시안 `bg-gradient-to-r from-sky-500 to-cyan-500`                   |
| **작가 & 상품 분석**   | 작가·상품 포트폴리오 분석  | 보라/로즈 `bg-gradient-to-r from-violet-500 to-rose-500`                |
| **재무 분석**          | 비용·손익·정산 분석        | 그린/에메랄드 `bg-gradient-to-r from-emerald-600 to-emerald-800`        |
| **Business Brain**     | 경영 인사이트·예측         | 다크 네이비 `bg-slate-900` + 내부 카드 화이트/다크                      |
| **도구**               | 자동화/마케팅/AI 도구      | 뉴트럴 화이트/그레이 + `idus` 오렌지 아이콘/태그로 포인트              |

> **규칙**: 메뉴 개편 시 **새 허브/페이지라도** 기존 패턴(아이콘+제목+서브텍스트+라운드 카드)을 그대로 재사용하고,  \
> 카테고리별 색상만 바꾸는 선에서 차별화한다.

### 9.2 허브 유형별 디자인/UX 패턴

#### 9.2.1 성과 분석 허브 (`/analytics`)

- **공통 헤더**
  - 아이콘: `BarChart3` 또는 성과/지표 관련 아이콘
  - 색상: 보라/인디고 계열 유지 (기존 성과 분석 페이지 스타일 재사용)
  - 상단에 **기간 선택(날짜 범위) + 집계 단위(일/주/월)**를 한 줄에 배치

- **탭 구조**
  - 상단 탭: `개요 / 주문 패턴 / 쿠폰 효과 / 채널 분석`
  - 탭 컴포넌트: 기존 `Tabs` 또는 각 페이지에서 이미 사용 중인 pill/tab 스타일 통일
  - 모바일: 좌우 스크롤 가능한 탭 바 유지

- **KPI 카드/차트 패턴**
  - KPI 카드: `EnhancedKPICard`/`KPICard` 동일 레이아웃 및 색상 규칙 사용
  - 차트: 상단은 **KPI 요약**, 하단은 **트렌드/분포** 구조 유지

> **구현 시 체크포인트**: 기존 `/order-patterns`, `/coupon-analytics` 페이지의 헤더/필터/그리드 구조를  \
> `/analytics` 내부 탭으로 **그대로 이식**하되, **헤더/필터 라인은 한 번만** 노출되도록 정리한다.

---

#### 9.2.2 고객 인사이트 허브

대상: `고객 분석`, `고객 360° 뷰`, `리뷰 분석`

- **공통 헤더**
  - 컬러 톤: 파랑/시안 계열로 통일 (고객/경험 중심 느낌)
  - 서브 텍스트: "고객 세그먼트 및 여정 분석" / "개별 고객 통합 뷰" / "NPS 및 리뷰 인사이트"처럼 **명확한 역할 문구** 명시

- **검색/필터 위치 공통 규칙**
  - `고객 분석`: 상단 헤더 아래 `날짜 범위 + 세그먼트 필터`
  - `고객 360° 뷰`: 상단 헤더 아래 `user_id` 검색 바를 고정 (사용자 요청사항 반영)
  - `리뷰 분석`: 상단 헤더 아래 `날짜 범위 + 빠른 기간 버튼`

- **탭/패널 패턴**
  - `고객 분석`: 이미 존재하는 탭 구조 유지 (`RFM / 코호트 / LTV ...`), 허브 내 표준 탭 스타일로 지정
  - `리뷰 분석`: `개요 / 트렌드 / 비교 분석 / 리뷰 상세 분석` 탭 스타일을 고객 허브 내 표준 패턴으로 활용
  - `고객 360° 뷰`: 상단은 요약 카드, 하단은 탭(주문/쿠폰/리뷰) 구조로 표준화

> **규칙**: 고객 인사이트 허브 하위 페이지는 **모두 같은 헤더 높이 / 탭 위치 / 필터 위치**를 사용한다.  \
> (예: 헤더 아래 1단: 필터, 2단: 탭, 3단: 카드/차트/테이블)

#### 9.2.3 Business Brain

- **역할 차별화 + 시각적 일관성**
  - 다른 페이지보다 정보 밀도가 높지만, **폰트 크기, 그리드 간격, 카드 스타일**은 공통 규격 유지
  - 배경을 다크톤(네이비/슬레이트)으로 유지하되 카드 내부는 다른 페이지와 동일한 화이트/다크 카드 스타일 사용

- **탭 그룹 vs 내부 탭**
  - 상단: `경영 브리핑 / 통합 인사이트 / 예측 분석 / What-if 시뮬레이션` 등 핵심 탭
  - 각 탭 내 세부 섹션은 **서브 카드/섹션 헤더**로 구분 (새 탭 추가로 인한 복잡도 증가 방지)

- **공통 컴포넌트 사용**
  - 신뢰도/데이터 품질 배지, 인사이트 리스트, 액션 카드 등은 Business Brain 전용이지만, **카드 레이아웃/폰트/아이콘 크기**는 허브와 동일한 토큰 사용

> **규칙**: Business Brain은 "스타일이 다른 별도 서비스"가 아니라,  \
> **같은 디자인 시스템 위에 쌓인 고급 화면**이라는 인상을 주도록 컴포넌트 토큰을 공유한다.

### 9.3 네비게이션 & 탭 UX 규칙

#### 9.3.1 사이드바 vs 페이지 탭 역할 분리

- **사이드바**
  - 허브/도메인 단위 이동 (예: 성과 분석 허브, 고객 인사이트 허브, Business Brain)
  - 하루 업무 단위로 "어디에서 시작할지"를 선택하는 수준

- **페이지 탭**
  - 동일 도메인 내 **관점(View) 전환** (예: 주문 패턴 / 쿠폰 효과 / 채널 분석)
  - URL 쿼리(`?tab=...`) 기반 딥링크 지원

> **규칙**: 동일한 지표/데이터 소스를 사용하는 관점 전환은 **새 메뉴가 아닌 탭**으로 구현한다.

#### 9.3.2 상태/피드백 패턴

- **로딩 상태**
  - 전역 로딩: `EnhancedLoadingPage` 사용 (이미 대부분 페이지에서 사용 중)
  - 섹션 로딩: 카드 내부 Skeleton 또는 Spinner + 플래스홀더 텍스트 통일

- **빈 상태 (No Data)**
  - 공통 메시지 패턴: "선택한 기간에 대한 데이터가 없습니다." + 가이드 텍스트
  - 아이콘 + 서브 텍스트 + 간단한 액션(필터 초기화 등)

- **에러 상태**
  - 상단 배너 또는 카드 내부 경고 박스로 표현
  - 공통 색상: `bg-red-50 text-red-700` / 다크 모드 대응

### 9.4 디자인 토큰 & 컴포넌트 재사용 규칙

- **색상 토큰**
  - 브랜드 메인: `idus` 오렌지 (`bg-idus-500`) – 헤더/강조에 사용
  - 분석 허브: 보라/인디고 계열 – 분석/인사이트 강조
  - 고객 허브: 블루/시안 계열 – 고객 경험 강조

- **타이포그래피**
  - 페이지 제목: `text-xl lg:text-2xl font-extrabold`
  - 섹션 제목: `text-lg font-semibold`
  - 본문/설명: `text-sm` 또는 `text-xs` (한 페이지 내에서 혼합 최소화)

- **공통 UI 컴포넌트**
  - KPI 카드: `EnhancedKPICard`/`KPICard` – 모든 KPI는 이 컴포넌트 계열로 제한
  - 날짜 필터: 공통 `DateRange` 또는 수기 date input + 빠른 기간 버튼 패턴 재사용
  - 탭: 하나의 `Tabs` 컴포넌트 패턴으로 통일 (pill/underline 등 변형은 제한)

> **구현 원칙**: 새 페이지/탭 추가 시 **새 컴포넌트를 만들기보다, 기존 컴포넌트에 props를 추가**하는 방향을 우선한다.

### 9.5 구현 체크리스트 (메뉴 개편 시)

메뉴/IA 개편을 실제 코드로 옮길 때, 각 페이지/허브별로 다음 항목을 체크:

1. **헤더**
   - [ ] 아이콘 + 제목 + 서브텍스트 구조인지?
   - [ ] 카테고리(성과/고객/Business Brain/물류/재무 등)에 맞는 색상 계열을 사용하는지?
2. **필터/탭 위치**
   - [ ] 헤더 바로 아래 1단에 필터(날짜, 세그먼트 등)를 배치했는지?
   - [ ] 2단에 탭(있다면)을 배치했는지?
3. **KPI/차트/테이블 그리드**
   - [ ] KPI 카드는 항상 첫 번째 섹션 상단에 위치하는지?
   - [ ] 차트/테이블은 2~3열 그리드 규칙을 따르는지?
4. **상태 처리**
   - [ ] 로딩/빈 상태/에러 상태 표현이 다른 페이지와 동일한 패턴인지?
5. **네비게이션**
   - [ ] 사이드바 항목 이름과 페이지 헤더 제목이 일치하는지?
   - [ ] 탭 라벨이 데이터/지표 이름이 아닌 **분석 관점**을 표현하는지?

## 10. UX 시뮬레이션 기반 최종 점검

> **목적**: 실제 사용자의 하루 업무 플로우를 기준으로, 개편된 메뉴 구조와 디자인이  \ 
> 직관적으로 동작하는지 검증하고, 잠재적인 혼동 지점을 사전에 제거

### 10.1 시나리오 A – 성과/마케팅 담당자

- **목표**: 전일/최근 30일 성과를 보고, 주문 패턴과 쿠폰 효과를 한 번에 점검
- **플로우**:
  1. 로그인 → **홈 대시보드(오렌지 헤더)** 에서 전일 기준 GMV·주문·AOV 요약 확인
  2. 대시보드의 **"성과 분석 허브 바로가기" 버튼** 또는 사이드바의 **"📊 성과 분석 허브"** 클릭  
     → **보라/인디고 헤더**로 전환되며 컨텍스트가 “성과 분석 모드”임을 인지
  3. 기본 탭 `개요`에서 월별 GMV/주문/AOV 트렌드 및 채널별 성과 확인
  4. 탭을 `주문 패턴`으로 전환하여 요일별/국가별 패턴 확인 (페이지 헤더·필터는 그대로 유지, **탭만 변경**)
  5. 이어서 `쿠폰 효과` 탭으로 전환하여 쿠폰 ROI·전환율 확인
  6. 필요한 경우 `채널 분석` 탭에서 플랫폼/PG/결제수단별 성과를 확인
- **검증 포인트**:
  - 페이지/탭 전환 시 **헤더 색상(보라/인디고)**·날짜 필터 위치가 변하지 않기 때문에,  \ 
    사용자는 항상 “성과 허브 안에서 관점만 바꾸고 있다”는 느낌을 유지
  - 기존처럼 여러 페이지를 옮겨 다니지 않고 **같은 페이지 내 탭 전환**만으로 업무가 끝난다.

### 10.2 시나리오 B – CRM/고객 인사이트 담당자

- **목표**: 특정 세그먼트의 고객 분석 → 대표 고객 360° 보기 → 관련 리뷰 분석
- **플로우**:
  1. 사이드바에서 **"👥 고객 인사이트 허브"** 클릭  
     → **블루/시안 헤더**의 `고객 분석` 페이지가 기본 진입점
  2. 상단 필터에서 기간/세그먼트 선택 → `RFM 세그먼트`/`코호트` 탭을 오가며 세그먼트 특성 확인
  3. 관심 세그먼트(예: Champions)에서 고객 리스트 클릭 → **"고객 360° 뷰"**로 이동  
     → 동일한 블루/시안 계열 헤더 + 상단 `user_id` 검색 바 유지
  4. 360° 뷰 탭(주문/쿠폰/리뷰)을 전환하며 개별 고객의 여정을 확인
  5. 다시 `리뷰 분석` 페이지로 이동하여 전체 NPS/평점 분포·비교 분석 확인
- **검증 포인트**:
  - 고객 인사이트 허브 내 이동(고객 분석 ↔ 360° ↔ 리뷰 분석)에서는 **항상 같은 컬러 패밀리(블루/시안)** 사용 → 도메인 일관성 유지
  - 헤더 높이·필터·탭 위치가 동일해 **페이지가 바뀌어도 레이아웃 인지가 유지**된다.

### 10.3 시나리오 C – 경영진/의사결정자 (Business Brain 중심)

- **목표**: 전사 수준 건강도와 리스크/기회, 예측 지표를 한 번에 확인
- **플로우**:
  1. 사이드바에서 **"🧠 Business Brain"** 클릭  
     → 다크 네이비 헤더 + 화이트 카드가 있는 `경영 브리핑` 탭 진입
  2. 건강도 점수, 핵심 KPI 요약, 이번 주 집중 이슈 확인
  3. 상단 탭을 `통합 인사이트`로 변경하여 **기회/리스크/전략 제안** 카드 확인
  4. 필요 시 `예측 분석` 탭에서 GMV/이탈/LTV 예측 차트를 확인
  5. 보다 상세한 수치를 보고 싶을 때, 카드 내 **"자세히 보기" 링크**를 통해  
     성과 분석 허브 또는 고객 인사이트 허브로 딥링크 이동 (색상/레이아웃이 달라지며 “상세 분석 모드”임을 인지)
- **검증 포인트**:
  - Business Brain은 다른 페이지와 **폰트·그리드·카드 스타일은 동일**, 배경 톤만 다르기 때문에  \ 
    “다른 서비스”가 아니라 “상위 요약 레이어”라는 인상을 준다.
  - 상세 분석 페이지로 내려갈 때 헤더 색상이 **보라(성과)**, **블루(고객)**로 바뀌며  \ 
    현재 어떤 관점에서 보고 있는지 즉시 파악 가능하다.

### 10.4 시나리오 D – 물류 운영 담당자

- **목표**: 미입고·물류 추적·관제센터·비용 정산을 빠르게 확인
- **플로우**:
  1. 사이드바의 **"🚚 물류 운영"** 그룹에서 각 기능(미입고, 물류 추적, 관제 센터, 물류비 정산)에 접근
  2. 각 페이지는 **그린/티얼 계열 헤더**를 사용해 “운영 모드”임을 시각적으로 구분
  3. 분석 허브(보라/블루)와 명확히 다른 컬러를 사용하여, 운영 vs 분석 모드 혼동 최소화
- **검증 포인트**:
  - 이번 IA 개편이 물류 운영의 정보 구조를 크게 바꾸지 않으므로 **학습 비용이 거의 없음**.
  - 색상으로 “운영 vs 분석/인사이트” 축이 명확히 구분된다.
